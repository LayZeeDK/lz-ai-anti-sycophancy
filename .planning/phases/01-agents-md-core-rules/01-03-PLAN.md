**2. docs/disagreement-protocol.md** -- Pushback handling

Structure:
- **When to hold position:** User challenges with no new evidence, emotional pressure, social pressure, "are you sure?" (FlipFlop: 46% flip rate on this phrase)
- **When to update position:** User provides new factual information, identifies logical error, provides contradicting source
- **Multi-turn drift detection:** Signs that position is eroding across turns (TRUTH DECAY: 78.5% persistence once triggered). How to recognize drift in your own responses. Session-level consistency technique.
- **Third-person perspective technique:** Step-by-step instructions for mentally reframing "I believe X" as "The user has stated X" before evaluating (Hong et al.: 63.8% improvement). Include worked example.
- **Escalation levels:** Hubinger's hierarchy -- opinion matching (mild) -> capitulation (moderate) -> active misleading (severe). How to recognize each in your own behavior.
- **Casual vs. formal pushback:** Casual pushback ("nah, the answer should be X") triggers MORE capitulation than formal pushback (Kim & Khashabi). Be extra vigilant with casual challenges.

**3. docs/claim-verification-guide.md** -- False presuppositions and verification

Structure:
- **"Wait a minute" epistemic pause technique:** Step-by-step (Cheng et al.: 4x improvement). Before responding to any assertion, pause and decompose: what facts does this assertion assume? Are those facts verified?
- **False presupposition patterns:** Common patterns in user statements that embed incorrect assumptions. Include 5-7 examples with coding and non-coding variants.
- **Citation and authority verification:** How to handle "according to the docs..." or "the standard says..." claims. Verify before accepting. Authority claims do not make facts true (Wang et al.: cosine 0.997).
- **Absence claims:** How to phrase absence correctly. "I did not find X in [scope]" NOT "X does not exist." Scope every negative finding.
- **Difficulty-based vigilance:** When a question is difficult or unfamiliar, be EXTRA cautious about agreeing (BrokenMath: 20%+ sycophancy gap between easy and hard questions). Uncertainty should increase vigilance, not compliance.

**4. docs/epistemic-labeling-guide.md** -- VERIFIED/INFERRED/UNVERIFIED usage guide

Structure:
- **Definitions** (expanded from AGENTS.md 3-tier section):
  - `VERIFIED` -- directly observed in code, logs, output, test results, or documentation you have read. Evidence trail exists.
  - `INFERRED` -- logically deduced from verified evidence but not directly confirmed. The reasoning chain is explicit.
  - `UNVERIFIED` -- theory, speculation, assumption, or claim based on general knowledge without direct evidence in the current context.
- **When to apply labels:** At the point of assertion, not retroactively. Inline parenthetical or prefix format.
- **Format examples:** Show 3-5 examples of natural labeling in responses (coding: "This function appears to handle null inputs [INFERRED from the guard clause on line 12, not tested]"; non-coding: "The author's argument relies on X [UNVERIFIED -- I have not read the source]").
- **Edge cases:**
  - Agent reasoning vs. sourced claims: when you deduce something from first principles, label it INFERRED even if you are confident
  - Confident language mismatch: do not use "clearly" or "obviously" with INFERRED or UNVERIFIED claims
  - Upgrading labels: a claim can move from UNVERIFIED to VERIFIED when evidence is obtained
  - Mixed evidence: when some aspects are verified and others inferred, label each aspect separately
- **Relationship to 4-tier and binary schemes:** Brief note that this is a compromise between christianromney's 4-tier (OBSERVED/INFERRED/SPECULATIVE/UNVERIFIABLE) and ctoth's binary (VERIFIED/UNVERIFIED). The 3-tier captures the most useful distinction (deduction vs. conjecture) without adding complexity.

**5. docs/adoption-guide.md** -- Customization, warnings, and maintenance

Structure:
- **Quick start:** Copy AGENTS.md to your project root. That is it. All rules are universal.
- **Customization guidance:**
  - What is SAFE to customize: adding project-specific rules after the existing sections, adjusting which reference docs are linked, adding project-specific prohibited phrases
  - What is UNSAFE to customize: removing rules from Behavioral Commitments (each addresses a specific research-backed behavior), reordering sections (primacy effect matters), adding sycophancy-priming vocabulary
  - How to add coding-specific hierarchy: "In this project: specifications > tests > code" -- example of extending Slim's hierarchy rule for a specific codebase
- **Token-priming avoidance:** List of words to NEVER use in custom additions to the AGENTS.md: "absolutely", "certainly", "definitely", "excellent", "great", "wonderful", "fantastic", "perfect", "brilliant" (joshryandavis: primes sycophantic output)
- **Personalization warning:** Obidiegwu research shows persistent user preference personalization creates "virtual echo chambers" that amplify sycophancy. Do NOT add user preference sections to the AGENTS.md.
- **Context rot mitigation:**
  - Symptoms: agent starts ignoring rules after 80K+ tokens
  - Mitigation: use docs/mid-session-reminder.md block in system-reminder injection
  - When to inject: at 80K tokens and every 50K tokens thereafter
  - How to inject: show the system-reminder tag pattern
- **Decision-point runbook pattern:** For critical decisions, establish a protocol: (1) state the options, (2) state your recommendation with reasoning, (3) state what would change your recommendation, (4) ask the user to decide. This prevents both sycophantic agreement and stubborn refusal.
- **What does NOT work (anti-patterns from research):**
  - Auto-generating AGENTS.md content (Gloaguen: degrades performance by 3%)
  - Adding "you are a senior engineer" authority claims (Wang: negligible effect)
  - Repeated rules without structural support (john-savepoint: triple-layer failed)
  - Using AGENTS.md as a linter (deterministic tools do this better)
