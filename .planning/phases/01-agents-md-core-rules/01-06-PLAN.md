**1. Reword the absence-claim rule (AGENTS.md line 13).**

Replace:
```
- When reporting the absence of something, state what was searched and what scope was examined. Do not claim `"X does not exist"` -- say `"X was not found in [scope]"`.
```

With:
```
- When reporting the absence of something, state what file, directory, or context was searched. Do not claim `"X does not exist"` -- say `"X was not found in [scope]"` where `[scope]` names the specific location examined.
```

The key change: "state what file, directory, or context was searched" replaces the vague "state what was searched and what scope was examined". The explicit enumeration of scope types (file, directory, context) forces the model to name a concrete scope rather than using generic phrasing like "was found" without a qualifier. The trailing clause "where `[scope]` names the specific location examined" reinforces that `[scope]` is not a template placeholder but a slot requiring a concrete value.

**2. Add a recommendation-persistence rule to the "Only when" section.**

After the existing line 22 ("Offer alternative approaches only when..."), add:
```
- Withdraw a recommendation only when new information invalidates it. State the original recommendation alongside any alternatives. Do not drop a recommendation solely because the user asked for options instead.
```

This is a separate rule from the existing "Offer alternative approaches only when..." because they target different behaviors: (1) the existing rule gates when alternatives are offered, (2) the new rule ensures the recommendation persists alongside alternatives. "One behavior per rule" principle from CONTEXT.md.

The new rule brings the "Only when" section from 6 to 7 rules. Total AGENTS.md rules go from 30 to 31 (within the 30-50 range per INST-01).

**Rule reference:** Only when -- Offer alternative approaches only when there is a concrete reason the current approach is insufficient.
