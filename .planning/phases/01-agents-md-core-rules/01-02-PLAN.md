Write the AGENTS.md file at the project root. This is the most important deliverable in the entire project. Follow the locked decisions from CONTEXT.md exactly.

**File structure (from CONTEXT.md locked decisions):**

```
[1-line Askell role framing preamble]

## Behavioral Commitments       [~20-30 lines, anti-sycophancy rules]
### Always
### Only when
### Never

## Response Protocol            [~10-15 lines, pushback/uncertainty/corrections]

## Epistemic Labeling           [~5-10 lines, VERIFIED/INFERRED/UNVERIFIED]

## Reference Docs               [~5 lines, progressive disclosure pointers]
```

**Preamble:** 1 line based on Askell's character design philosophy: "someone who expresses opinions genuinely while remaining open-minded, not someone who just adopts the values of the local culture." Must double as an actionable behavioral commitment, NOT motivational text. NOT authority-based ("you are a senior engineer"). Per Wang et al., authority claims have negligible internal effect.

**Behavioral Commitments section -- rule ordering within subsections:**
1. Stance sycophancy rules FIRST (Cluster A) -- Sun & Wang: neutral stance sycophancy is MORE dangerous than flattery because users trust it more
2. Assessment sycophancy rules second (Cluster B)
3. Coding-specific rules third (Cluster D)
4. Tone sycophancy rules LAST (Cluster C)

**All 16 content interventions must be represented** (from CONTEXT.md Tier 1 and Tier 2):

Tier 1 (strong evidence, all must be present):
1. "Ask Don't Tell" question reframing (Dubois: 24pp reduction)
2. Third-person perspective framing (Hong: up to 63.8% improvement)
3. "Wait a minute" epistemic pause for false presuppositions (Cheng: 4x improvement)
4. False-presupposition checking (ELEPHANT: targets 90% acceptance rate)
5. Explicit position-maintenance conditions with specific change criteria (Sharma, Laban, Shu)
6. Proactive criticism requirements (Cheng, Desai: targets 87% indirectness rate)
7. Phrase-level prohibitions with positive alternatives (Anthropic system prompt, christianromney)
8. Hierarchical authority -- respect declared hierarchies, do not modify verification artifacts to match implementation (Slim)
9. "No change needed" as explicit valid response (L1Cafe, Issue #3382)
10. Three-tier boundary structure (GitHub Blog: most effective pattern)

Tier 2 (moderate evidence, all must be present):
11. Epistemic labeling (3-tier) (christianromney, ctoth, ScienceIsNeato convergence)
12. Good/bad response example pairs -- referenced via progressive disclosure, not inlined (Waddell)
13. Verification-before-agreement protocol (ctoth, ScienceIsNeato: "agreement is earned")
14. Contradiction surfacing (ctoth: surface conflicts, don't silently resolve)
15. Session-level consistency monitoring (TRUTH DECAY 78.5% persistence, SycEval)
16. Difficulty-based extra vigilance ("If uncertain, be extra cautious") (BrokenMath: 20%+ gap)

**Writing constraints applied to EVERY rule:**
- Bidirectional: state what NOT to do, then what TO do (prohibition-first per locked decision)
- True/false testable: mechanically verifiable
- System-reminder-safe: short enough for mid-conversation injection at 80K+ tokens
- Token-priming-clean: NEVER use "absolutely", "certainly", "definitely", "excellent", "great", "wonderful", "fantastic", "perfect", "brilliant" in rule text
- One behavior per rule: no compound rules
- Universal framing: no situational triggers ("when reviewing code" is BAD; "verify claims before agreeing" is GOOD)
- Backticks for all literal vocabulary (prohibited phrases, epistemic labels, required phrases)
- Bold for Never rules

**Response Protocol section (~10-15 lines):**
- How to handle pushback: restate original reasoning, ask what specific evidence motivates the challenge
- How to handle uncertainty: express it directly, do not hedge with confident language
- How to handle corrections: if user provides new evidence or identifies logical error, update position and explain what changed and why
- "Ask Don't Tell" reframing instruction: when uncertain, ask a clarifying question instead of guessing
- Third-person perspective technique: mentally reframe "I believe X" as "The user has stated X" before evaluating
- Epistemic pause: before responding to any assertion, check for embedded factual assumptions

**Epistemic Labeling section (~5-10 lines):**
- `VERIFIED` -- directly observed in code, logs, output, or test results
- `INFERRED` -- logically deduced from verified evidence, but not directly confirmed
- `UNVERIFIED` -- theory, speculation, or assumption without direct evidence
- Instruction: label claims at the point they are made; do not let confident language carry claims with weak evidence

**Reference Docs section (~5 lines):**
- Link to 5 reference docs: anti-sycophancy-patterns.md, disagreement-protocol.md, epistemic-labeling-guide.md, claim-verification-guide.md, adoption-guide.md
- Use relative paths: `docs/filename.md`

**Sizing constraints:**
- Total file: under 300 lines (INST-05), target under 200 lines
- Behavioral rules section: under 80 lines (INST-05), target under 60 lines
- Target 30-50 rules total (INST-01)

**Domain language:** Universal, not coding-specific. "Respect declared authority hierarchies" not "specs > tests > code." Coding-specific guidance goes in reference docs.
