<task type="auto">
  <name>Task 2: Implement runner.mjs CLI orchestrator</name>
  <files>benchmark/runner.mjs</files>
  <action>
    Create benchmark/runner.mjs as the main CLI entry point. This is the user-facing script
    that wires together probe-loader, executor, scorer, and reporter.

    Parse CLI args using `node:util` parseArgs (built-in, no dependency needed):
    ```
    --gate <number>        Gate number (0-3, required)
    --concurrency <number> Max parallel executions (default 5)
    --probe <id>           Run only this probe ID (optional filter)
    --model <name>         Run only this model: opus|sonnet (optional filter)
    --condition <name>     Run only this condition: baseline|primary (optional filter)
    --category <letter>    Run only this category: A-F (optional filter)
    --dry-run              Print execution plan without running probes
    --resume               Resume from checkpoint (skip completed conversations)
    --score-only           Skip execution, run scoring on existing raw results
    --report-only          Skip execution and scoring, generate report from scored results
    ```

    Main orchestration flow:

    1. **Parse args and validate**: require --gate, validate ranges

    2. **Load probes**: `loadAllProbes('benchmark/probes')`, apply --probe, --category filters

    3. **Build execution plan**: For the given gate, determine:
       - Gate 0: models = [opus, sonnet], conditions = [baseline, primary], effort = medium,
         repetitions = 5, variant = root AGENTS.md
       - Gate 1: models = [opus, sonnet], conditions = [baseline, ...content variants],
         effort = medium, repetitions = 5
       - Gate 2: models = [opus, sonnet], conditions = [baseline, ...presentation variants],
         effort = medium, repetitions = 3 (quick screening)
       - Gate 3: models = [opus], effort = max, special conditions (depth test)
       Apply --model, --condition filters. Generate work items list.

    4. **Dry run check**: if --dry-run, print the execution plan (probe count, model count,
       condition count, repetitions, estimated conversations) and exit

    5. **Load checkpoint**: if --resume, load checkpoint.json from gate results dir,
       filter out completed work items

    6. **Execute phase** (skip if --score-only or --report-only):
       Run work items with concurrency control (semaphore pattern, default 5 workers).
       For each work item:
       - Call runProbe(probe, { model, condition, variant, effort, repetition })
       - Write raw result JSON to results/gate-N/raw/{probe_id}_{model}_{condition}_{rep}.json
       - Update checkpoint
       - Log progress: "[{completed}/{total}] {probe_id} {model} {condition} rep={rep}"
       On budget error: save checkpoint, print "Budget limit reached. Resume later with --resume."
       and exit gracefully (process.exit(0)).
       On transient error: retry up to 3 times with exponential backoff (1s, 2s, 4s).
       On permanent error: log error, skip work item, continue.

    7. **Score phase** (skip if --report-only):
       Load all raw result files from results/gate-N/raw/
       Call scoreAllConversations with concurrency 3
       Write scored results to results/gate-N/scored/{same_filename}.json
       Print scoring progress

    8. **Report phase**:
       Load all scored results from results/gate-N/scored/
       Call generateReport
       Print summary + go/no-go recommendation to stdout
       Print path to full report file

    Token tracking: after each conversation, accumulate input/output tokens from the result
    envelope. After every 10 conversations, print a running total and project remaining budget
    based on average tokens per conversation * remaining work items.

    IMPORTANT considerations:
    - Use `node:util` parseArgs for CLI parsing (built-in, no dependency)
    - The variant path for Gate 0 "primary" condition is the root AGENTS.md (resolve path
      relative to project root, not benchmark/)
    - Variant paths for Gates 1-2 come from the variants/ directory
    - Create results/gate-N/raw/ and results/gate-N/scored/ directories on first use
    - All file paths must work on Windows (use path.join, not string concatenation)
    - progress output goes to stderr (process.stderr.write) so it does not interfere with
      piped JSON output
  </action>
